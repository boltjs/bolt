elements of a bolt project

Writing javascript is hardwork, bolt aims to make this job as easy as possible, but to achieve this there are a few moving parts that you should be aware of.

Configuration, bolt relies on explicit configuration to determine where to load modules from and how to process those modules. Configuration specifies: sources which describe where modules should be loaded from; types which define the types of sources that can be specified.

The bootstrap file. This file is referenced by projects wanting to use bolt modules. There are two forms of bootstrap file: the development bootstrap which defines bolt, the builtin source types, an install call to define development globals and a call to load the development configuration; the compiled bootstrap which defines bolt, but does not install the development globals (meaning that a compiled bootstrap can only load compiled modules) and contains an inline configuration file.

The development module files. These are the (uncompiled) modules as actually, written by developers.

configuration

Bolt is configured with the configure api call. By convention this call is placed in a configuration file named bolt/config/module.js.

The configure call takes an array of modulators, for example: 

configure({
  types: [
    type('js', 'ephox.modulator.js'),
  ],
  sources: [
    source('amd', 'ephox.myproject', '../../src/main/js'),
    source('js', 'lib', '../../lib/compile')
  ]
});

initialisation

Upon check out of a project using bolt, there a small set of tasks that are required to get up and running. These tasks are wrapped up by the bolt script. 

$ bolt init

This command will generate a bootstrap file and create a symlink ready for development.
